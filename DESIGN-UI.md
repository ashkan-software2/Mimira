# DESIGN-UI — Mimira Admin Visual Design System

Generated by `/design-consultation` on 2026-05-25. Branch: `main`. Status: APPROVED.

Companion to `DESIGN.md` (product + architecture). This file is the single source of truth for visual tokens, typography, color, spacing, motion, and component primitives. Always read this before making any visual or UI decision in the Mimira admin app.

For the full product context, target users, channel constraints, and screen inventory, see `DESIGN.md` (especially the Design Review appendix). This file picks up where that one stopped: at "what should it look like."

---

## Aesthetic Direction

**Direction:** Quiet Modernist (cool).
**Decoration level:** Minimal. Type and whitespace do the work.
**Mood:** Calm and quiet, never overwhelming. Reads as serious software for serious work. The UI disappears into the workflow.
**Reference points:** ChatGPT.com, Apple.com.

**The polestar:** *Even at peak Line volume, the UI feels under control.*

Every design decision in this file serves that single sentence. A choice that makes the UI louder, busier, or more decorative — no matter how "modern" it looks — is wrong.

**What this is not:** This is the admin/cockpit aesthetic. The customer-facing surface is Line itself — Mimira's warmth lives in her *message copy*, not in chrome. The admin is the instrument, not the hug.

## Typography

**One family across Latin and Thai.** This is non-negotiable. Plex Sans Thai Looped is purpose-paired with Plex Sans Latin — one of the only Thai-Latin font pairs designed as a coordinated unit.

| Role | Family | Weights |
|---|---|---|
| Display, Body, UI (Latin) | **IBM Plex Sans** | 400 · 500 · 600 |
| Display, Body, UI (Thai) | **IBM Plex Sans Thai Looped** | 400 · 500 · 600 |
| Data / Tables / Numerics | IBM Plex Sans + `font-variant-numeric: tabular-nums` | 400 · 500 |
| Code (audit log, JSON viewers) | **IBM Plex Mono** | 400 · 500 |

**Loading:** Google Fonts CSS, subset to Latin + Thai. Display swap. Approx 110 KB total at the three weights.

```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link href="https://fonts.googleapis.com/css2?family=IBM+Plex+Sans:wght@400;500;600&family=IBM+Plex+Sans+Thai+Looped:wght@400;500;600&family=IBM+Plex+Mono:wght@400;500&display=swap" rel="stylesheet">
```

**Stack:**
```css
font-family: 'IBM Plex Sans', 'IBM Plex Sans Thai Looped', system-ui, sans-serif;
```
Order matters: Latin first, Thai second. Browsers fall through per-glyph for mixed-script lines.

**Type scale** (modular 1.2 ratio):

| Token | Size / line | Letter-spacing | Weight | Use |
|---|---|---|---|---|
| `text-xs` | 12 / 16 | 0 | 400 | meta, timestamps, badges |
| `text-sm` | 14 / 20 | 0 | 400 | compact table rows |
| `text-base` | 16 / 24 | 0 | 400 | body (Thai needs 16 minimum) |
| `text-md` | 18 / 28 | 0 | 400 | lead paragraphs |
| `text-lg` | 20 / 30 | −0.01em | 500 | h3 |
| `text-xl` | 24 / 32 | −0.015em | 600 | h2 |
| `text-2xl` | 32 / 40 | −0.02em | 600 | section headers |
| `text-3xl` | 40 / 44 | −0.025em | 600 | display (rare in admin) |

**Rules:**
- Body text **minimum 16 px**. Thai script is unreadable at 14 px for casual register.
- Tabular numerics on every numeric column: `font-variant-numeric: tabular-nums`. Push-quota, message counts, timestamps, prices.
- Weights stop at 600. Never use 700+ — too much weight on dense screens.
- Display tracks tighten with size; body stays at default tracking.

### The AI accent rule (load-bearing)

`#0a7c7c` appears in **exactly three places** in v0:
1. The small `Mimira` badge inline with every AI-generated reply in the inbox thread.
2. The AI brain card in Settings (and its kill-switch toggle indicator).
3. Any future AI-only UI element that needs to teach the eye "this came from the model."

It does **not** appear on buttons, navigation, brand marks, charts, or decoration. Wherever you see teal, AI is at work. Breaking this rule kills the system's most useful piece of visual semantics.

**Base unit:** 4 px.
**Density:** Comfortable. Tighter than airy marketing sites; looser than data-grid utilitarianism.

| `space-sm` | 8 px | Inline gaps in rows |
| `space-md` | 12 px | Default vertical rhythm inside a card |
| `space-lg` | 16 px | Card padding bottom |
| `space-xl` | 24 px | Card padding, section gaps |
| `space-2xl` | 32 px | Between sections |
| `space-3xl` | 48 px | Page-level gaps in marketing/onboarding |
| `space-4xl` | 64 px | Hero rhythm |
| `space-5xl` | 96 px | Top of page, between major slabs |

**Containers:**
- Marketing/onboarding max width: `1280px`.
- Admin (Inbox, Knowledge, etc.): full-bleed three-column.

## Layout & Geometry

**Approach:** Grid-disciplined. Strict columns, predictable alignment.

**Border radius hierarchy** (no uniform bubbly radius):

| Element | Radius |
|---|---|
| Buttons, inputs, chips | 6 px |
| Cards, panels, message bubbles | 8 px |
| Modals, large surfaces | 12 px |
| Avatars, dots, pill badges | 9999 px (full) |

**Elevation:**
- Default: borders, not shadows. `--border` carries the visual hierarchy.
- Modal shadow (only place a shadow is allowed): `0 4px 20px rgba(0,0,0,0.08)`.
- No layered shadows. No glow effects. No inner shadows.

**Focus ring (every interactive element):**
```css
outline: none;
box-shadow: 0 0 0 2px var(--bg), 0 0 0 4px var(--link);
```
2 px offset from the element via a bg-colored inner ring, 2 px solid blue outer. Visible on all backgrounds. Never use `outline: none` without replacement.

**Inbox grid:** `grid-template-columns: 320px 1fr 280px` on desktop ≥ 1100 px. Right rail drops to a slide-out drawer at < 1100 px. Mobile (< 768 px): left rail and thread stack, right rail moves to a header-action drawer.

## Motion

**Approach:** Minimal-functional. Motion serves comprehension, not delight.

**Easing tokens:**
- Enter: `cubic-bezier(0.16, 1, 0.3, 1)` (smooth ease-out, "settle in")
- Exit: `cubic-bezier(0.7, 0, 0.84, 0)` (sharper ease-in, "snap out")
- Move: `cubic-bezier(0.45, 0, 0.55, 1)` (symmetric ease-in-out)

**Duration tokens:**
- `motion-micro`: 80 ms (hover, press, focus rings)
- `motion-short`: 160 ms (toasts, popovers, tab switches, knowledge-tree expand)
- `motion-medium`: 240 ms (modal entrance, new-escalation slide-in)
- `motion-long`: 400 ms (skeleton-to-content reveal only; nothing else)

**The three intentional motions in the admin** (per design-review appendix):
1. New escalation slides in from top of left rail · 240 ms enter · subtle amber pulse on the badge for 600 ms · fades to default.
2. Save toast: fade in 160 ms, hold 2.4 s, fade out 240 ms.
3. Knowledge-tree expand/collapse: 160 ms ease-out.

**Banned motion:**
- No bounce. No spring.
- No parallax.
- No scale-from-zero entrances.
- No page transition flourishes.
- No looping animations (skeletons excepted — but use a slow 1.6 s shimmer, not 800 ms).

**TopLoader — navigation progress bar** (component: `app/TopLoader.tsx`):
- A 3 px-high bar fixed to the very top of the viewport (`position: fixed; top/left/right: 0`), above everything (`z-index: 9999`), non-interactive (`pointer-events: none`).
- Fill color is the `--ai-accent` token (`#0a7c7c` light / `#2dd4bf` dark) with a soft glow (`box-shadow: 0 0 8px` at ~50% accent). This is the one place accent is used as a moving element.
- Motion: container opacity fades in/out over ~200 ms; the fill `width` animates 0 → 100% over ~200 ms ease. (Sits between `motion-short` and `motion-medium`; tokenize as a follow-up.) It trickles 8% → 88% while the route loads, then snaps to 100% and fades on arrival.
- **Not** a banned "page transition flourish": it is a functional progress affordance for slow navigations, not decorative page choreography. Nothing else on the page moves.

## Components — Primitives

### Buttons

Three variants. **No others.**

```css
.btn { padding: 9px 14px; border-radius: 6px; font-size: 14px; font-weight: 500; line-height: 1; border: 1px solid transparent; }
.btn-primary { background: var(--cta-fill); color: var(--cta-fg); border-color: var(--cta-fill); }
.btn-secondary { background: var(--surface); color: var(--fg); border-color: var(--border); }
.btn-ghost { background: transparent; color: var(--fg); }
```

Rules:
- Every page has at most **one** primary CTA in view at a time (excluding global "Send" composers).
- Secondary buttons sit next to primary CTAs only when a true alternative action exists.
- Ghost buttons for destructive-but-not-emphasized actions (Flag, Pause AI).
- Disabled: `opacity: 0.5; pointer-events: none;` — do not change color.

### Inputs

```css
.input { padding: 9px 12px; border-radius: 6px; border: 1px solid var(--border); background: var(--surface); font-size: 14px; }
.input:hover { border-color: var(--fg-muted); }
.input:focus { outline: none; border-color: var(--link); box-shadow: 0 0 0 3px color-mix(in srgb, var(--link) 18%, transparent); }
```

- Field-level validation errors render inline below the input in `--error`, 12 px. Never use global error banners for form fields.
- Required-field marker: small muted asterisk after the label. Not in the input.

### Badges

| Variant | Border + tint | Use |
|---|---|---|
| Needs attention | `--warning` | Any AI-side escalation (tripwire keyword match, low LLM confidence, judge-fail). One label, one color — staff doesn't triage by internal mechanism. |
| Staff requested | `--info` | Customer asked for human staff |
| Mimira | `--ai-accent` | Inline marker on every AI-generated reply |

The `Mimira` badge:
```html
<span class="badge badge-mimira">Mimira</span>
```
Renders ~10.5 px, uppercase letterforms, `--ai-accent` foreground on `--ai-accent-bg`. Single rule: every model-generated message gets one, every staff message does not.

### Alerts

Four stateful variants: success, warning, error, info. 8 px radius, soft tinted background, 1 px tinted border, 14 px body. Single colored 16 px dot icon on the left — no fancy iconography.

### Message bubbles (Inbox)

| Sender | Side | Surface | Border |
|---|---|---|---|
| Customer | left | `--surface` | `--border` |
| Mimira (AI) | left | `--surface` | `--border` + inline `Mimira` badge |
| Staff | right | `--cta-fill` | none, white text |

12 px radius. Max width 78% of thread column. Meta row below each bubble: 11 px tabular timestamp + optional source count for Mimira replies.

## Anti-Slop Hard Rules

These are blocking. A PR that ships any of these is wrong:

- **No purple, violet, or indigo anywhere.** Including hover states, gradients, and "fun" empty illustrations.
- **No gradients.** Not on buttons. Not on backgrounds. Not on accents. Not as decoration.
- **No 3-column feature-grid with icons in colored circles.** Not anywhere — onboarding, settings, knowledge empty state.
- **No centered hero copy in the admin app.** Only marketing/onboarding heroes may center (and only when there's a single point of focus).
- **No uniform bubbly border-radius (16 px everywhere).** Use the 6/8/12 hierarchy.
- **No decorative blobs, floating circles, wavy SVG dividers, animated background grids.**
- **No `system-ui` as the primary font.** Always load IBM Plex Sans. System fallback is for the first paint only.
- **No "Welcome to Mimira" or "Powered by AI" or "Built for clinics" hero copy.** Concrete one-sentence statements only.
- **No emoji as UI decoration.** Emoji is allowed only inside user-generated content (customer messages, Knowledge body, brand-voice samples).
- **No drop-shadows on cards.** Borders carry hierarchy. Shadow is reserved for modals.
- **No weight heavier than 600.** Including bold body text.

## How to evolve this system

- Visual change to one token → update this file's table + the CSS block, log a row in the Decisions Log.
- New component primitive → add it to Components with the same rigor as existing ones (variants, states, banned uses).
- A team member wants to break a rule (e.g., add a second accent color, ship a gradient) → open a discussion in PR comments referencing the specific rule. The polestar (`calm and quiet, never overwhelming`) is the test.
- Major visual direction shift → run `/design-consultation` again. Do not piecemeal.

For visual QA after shipping a change, run `/design-review`. It diffs the live site against this file's rules and flags violations.
