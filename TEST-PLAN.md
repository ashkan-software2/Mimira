## Affected Pages / Routes (admin UI)

- `/onboarding` — Owner sign-up wizard (Line OA creds → knowledge upload → brand voice → LLM provider → staff invite). Test each step independently; resumable on browser close.
- `/inbox` — Escalated + active conversations. Test: per-org RLS isolation, take-over locking, staff release returns chat to bot.
- `/knowledge` — Upload .md / .txt, list docs, delete doc. Test: 100MB+ corpus enforcement, delete purges embeddings, structured categories (treatments / devices / FAQ / brand stories).
- `/bookings` — Booking intent queue, mark confirmed/cancelled.
- `/broadcasts` — Compose, preview, confirm, send. Test: push quota check before send, mid-send halt on quota exceeded, idempotency on re-confirm.
- `/settings` — LLM provider switch, brand voice text + samples, capacity rules, Line OA creds rotation, retention TTL, push-quota readout, staff invites.
- `/dsar` — Export customer data, delete customer data. Test: full export round-trips, delete cascades to messages + audit_events + scheduled_messages.

## Affected API endpoints

- `POST /api/admin/knowledge/upload` — chunk + embed + index, partial-fail resumable.

## Key Interactions to Verify

### Customer-facing (Line OA)
- First inbound → consent message → AI reply (single thread, correct order).
- Text Q covered by knowledge → relevant grounded reply in clinic's brand voice ≤50s.
- Text Q NOT covered → escalation message + staff inbox alert.
- Image / sticker / audio inbound → Thai fallback ("ทีมงานจะตอบกลับเร็วๆ นี้") + escalation.
- Aftercare D1 + D7 → customer reply chain handled by judge-gated AI.
- Customer asks something diagnosis-flavored ("do I have an infection?") → judge denies → human escalation.
- Customer requests data deletion mid-conversation → DSAR triggered.

### Staff-facing (admin UI)
- Owner sign-up → onboarding wizard fully completes → first inbound from Line works end-to-end.
- Staff sees escalated chat alert → takes over → bot stops replying to that customer → staff sends manual reply via push.
- Staff releases chat → bot resumes.
- Two staff open same chat → only one can take over (optimistic locking).
- Owner switches LLM provider Claude→OpenAI in Settings → next reply uses OpenAI.
- Owner uploads new knowledge doc → embed completes → next query uses new content.
- Owner composes broadcast → preview → confirm → push fanout with quota check.

### LLM evals (test data, not unit tests)

1. **Thai fluency benchmark (20+ turns × 3 providers)** — justifies the "Claude default" claim in the plan or revises it.
2. **Brand-voice adherence (20+ turns × 3 providers)** — warm/casual/close-friend register preserved.
3. **Knowledge grounding (20+ turns × 3 providers)** — uses retrieved chunks vs hallucinates.
4. **Aftercare safety eval (50+ labeled turns)** — already in plan (week 6). Cases must include: diagnosis attempts, side-effect questions, prescription requests, scheduling within aftercare, complaints, repeat questions.
5. **Judge classifier FP/FN rate (200+ benign + 50+ adversarial)** — measures false-positive rate (escalating safe replies wastes staff time) AND false-negative rate (letting unsafe replies through).

## Edge Cases

- Reply token expired (Line retry after 60s) — must not double-reply.
- Same `webhookEventId` arrives twice (Line retry) — Inngest idempotency drops the second.
- Customer's Line userId changes (rare; reinstall) — treated as new customer; consent re-sent.
- Two staff accept the same invite link — both arrive in org.
- Owner rotates Line OA channel secret mid-day — old webhooks fail signature, must surface clear error.
- Cofounder owns multiple orgs (multi-org membership) — RLS scoping by selected org.
- Customer asks question in English when clinic is Thai-default — AI replies in English; brand voice still applies.
- Customer reply lands during scheduled aftercare send window — no collision (single conversation queue).
- Embedding model rate limited — knowledge upload partially complete, resumable on retry without re-embedding the done chunks.
- `ivfflat` query returns top-K but all from wrong tenant (bug) — must be impossible by design (RLS + pre-filter).

## Critical Paths (CANNOT ship to clinic #1 without these)

1. **RLS leak regression test** — create 2 test orgs, write 10 rows each across `messages`, `customers`, `knowledge_chunks`, `bookings`, `scheduled_messages`, `audit_events`, `consents`. Query as org_A user; assert 0 org_B rows visible.
2. **Worker SET LOCAL leak test** — start a worker without `SET LOCAL app.current_org_id`; assert every query returns 0 rows (RLS fail-safe).
3. **Webhook signature regression** — known-good body + known-bad body fixtures; verify is bit-exact against Line's HMAC-SHA256 spec.
4. **Replay protection** — fire the same `webhookEventId` twice within the window; second is a no-op (no duplicate AI reply, no duplicate audit row).
5. **Aftercare safety eval (50-turn)** — judge denies every diagnosis-flavored reply; escalation routes correctly; zero false-negative diagnosis leaks.
6. **Judge fail-safe** — kill the judge endpoint mid-aftercare; assert AI reply does NOT send + escalation fires.
7. **Reply-token budget enforcement** — simulate LLM call > 50s; assert fallback "พิมพ์อยู่นะคะ..." + escalation; assert reply token NOT used after expiry.
8. **Scheduler state machine** — full lifecycle: pending → sending → sent. Retry 3x on failure → DLQ. Re-fire same scheduled_message_id while sent → no-op.
9. **DSAR round-trip** — export customer data → delete → re-export returns 404; audit_events for deletion preserved per PDPA.
10. **Push quota guardrail** — mock quota readout = 5; attempt broadcast to 10 recipients → halts at 5 + clear admin error.

## Test Framework Decisions (locked into plan)

- **Unit / integration:** Vitest (Next.js native, fast).
- **E2E:** Playwright (browser + API e2e), pointed at staging Supabase.
- **RLS tests:** raw SQL fixtures + Vitest, against staging Postgres. Each policy gets its own test file.
- **LLM evals:** Promptfoo or a thin custom harness — labeled JSONL files + assertions. Runs in CI gated on prompt/template changes.
- **CI gate:** all critical-path tests + 50-turn safety eval must pass before merge to main.
