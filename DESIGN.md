
# Design: Yuna v0 — Smart Receptionist + Lite Ops for Thai Skin Clinics

Generated by /office-hours on 2026-05-25
Branch: main
Repo: ashkan-software2/Yuna
Status: APPROVED
Mode: Startup

## Problem Statement

Thai skin clinics use Line as their primary customer touchpoint. Inbound inquiries pile up overnight, staff burn out replying in casual Thai during the day, after-hours leads ghost, and the warm "close-girlfriend" register that Thai consumers respond to is impossible to maintain consistently across staff and shifts. Existing chatbot tools are robotic, English-first, or generic — they erode trust rather than build it.

Clinics need an AI that handles inquiries with brand-specific warm Thai voice, answers technical treatment questions from clinic-owned knowledge, escalates uncertain cases to staff, captures booking intent, runs D1/D7 aftercare followups, and lets the clinic broadcast simple promotions — all multi-tenant from day 1 because the cofounder brings clinic-network distribution.

## Demand Evidence

- One paying clinic client with terms agreed upfront. Decision-maker is bought in.
- Same client wants to become a cofounder and ship Yuna across their peer clinic network — a concrete distribution channel, not hypothetical interest.
- Founder reaction in-session ("this is big") followed by re-ranking priorities in real time when the cofounder context entered. Real signal, not founder cosplay.

## Status Quo

- Clinic staff manually reply to Line inquiries in Thai during business hours. Backlog overnight.
- Technical product questions (device differences, treatment specs) bottleneck on the most-experienced staff member, who is also booking appointments.
- Appointment booking happens via DM threads or external systems — varies per clinic, often no shared source of truth.
- Aftercare patient messages drown the same staff with repeat questions already answered pre-treatment.
- Promotions sent ad-hoc via Line OA broadcast tools, no per-customer interest tracking, no follow-through measurement.

## Target User & Narrowest Wedge

- **Primary buyer:** Clinic owner / GM at a Thai skin clinic.
- **Daily user:** Clinic staff (front desk, admin) using the Yuna admin UI; AI agent handles direct customer contact.
- **End consumer:** Thai customer messaging the clinic's Line OA about treatments.
- **Wedge:** Smart Receptionist on Line in TH/EN with clinic-specific warm voice, grounded on the clinic's uploaded knowledge, escalating uncertain cases to staff, capturing booking intent, sending D1/D7 aftercare followups, and supporting simple broadcast — multi-tenant from day 1.

## Constraints

- **Languages:** Thai (primary) + English. Architecture supports adding more later (CN, JP plausible next).
- **Channel:** Line first. Facebook, WhatsApp, Instagram later via a channel-adapter interface — no Line-only assumptions baked into the core.
- **Compliance:** Thai PDPA + medical safety. Detailed below.
- **Voice:** Per-clinic configurable brand voice. Default register for Thai market is warm, casual, close-friend.
- **LLM choice:** Per-clinic selectable from OpenAI / Anthropic / Google providers in the admin UI. Set once, locked until owner changes it. (Buyer-anxiety feature; the clinic owner explicitly asked for control over cost + vendor + trust. Keep.)
- **Hosting region:** Thailand is preferred, for example AWS Bangkok. Singapore is next (closest to Thailand for latency + aligned with PDPA data-residency expectations).
- **Knowledge format v0:** Text/Markdown upload, embedded into vector DB, retrieved on each chat turn.

### Line API Constraints (v0)

- **Webhook signature verification** required on every request (HMAC-SHA256 of body with channel secret). Reject unverified webhooks before any work.
- **Reply token expires in 60 seconds.** This forces async: ack the webhook immediately (< 1 sec), then process + reply via push or queued reply. Design assumes a queue, not in-request LLM calls.
- **Monthly push quotas** per Line OA (varies by clinic plan). Broadcast and D1/D7 aftercare consume push budget; the admin UI must surface remaining quota or the clinic will hit a cliff. v0: show current month's used/limit on Settings.
- **Inbound message types:** customers send images of skin conditions and stickers constantly. v0 = text only; image/sticker/audio messages get a polite Thai fallback ("ทีมงานจะตอบกลับเร็วๆ นี้") and route directly to staff escalation queue. Image understanding deferred.

### PDPA Compliance (v0)

- **Consent message** on first contact from a Line user — short Thai notice covering AI processing, sub-processors (the 3 LLM vendors), and the right to request data deletion. Sent automatically on first inbound; recorded in `consents` table.
- **DSAR endpoint** in admin UI for staff to export or delete a customer's data within 30 days.
- **Retention:** conversation messages TTL'd at 24 months by default, configurable per clinic.
- **Sub-processor disclosure** visible in admin UI (which LLM provider is active, what data leaves Singapore for which calls).
- **Per-clinic DPA template** in repo as `/legal/dpa-template.md` (not lawyer-vetted; placeholder for the founder's lawyer to fill).

### Medical Safety (non-negotiable)

- AI must never diagnose, prescribe, or estimate treatment outcomes. Aftercare flow is the highest-risk surface.
- **Three-layer guardrail:** (1) system prompt with explicit deny-list of behaviors, (2) keyword tripwires (TH/EN: "diagnose", "วินิจฉัย", "side effect", "ผลข้างเคียง", etc. unless are given in the dataset by the clinic) that hard-escalate to staff, (3) post-generation LLM-judge classifier on aftercare replies that audits before sending.
- **Audit log:** every aftercare reply persisted to `audit_events` table with input, output, guardrail decisions, classifier verdict. Reviewable by staff.
- **Eval set:** 50+ labeled aftercare turns (built with clinic #1's input) used as a regression test before week 6 ships.

## Premises (agreed)

1. Multi-tenant from day 1, `org_id` (clinic_id) on every table.
2. LLM provider = pluggable abstraction, per-clinic configuration (OpenAI / Anthropic / Google).
3. Hosting in Thailand if possible, or near Thailand (Singapore region) for latency + PDPA alignment.
4. Knowledge ingestion = text-based RAG via pgvector (this can be change later but for now can choose). No fine-tuning. No raw prompt-stuffing.
5. Booking v0 = capture booking intent in DB + admin UI queue. Staff confirms manually. No calendar integrations. Can have it for a client who pays for this integration.
6. Escalation v0 = staff sees flagged chats in admin inbox and can take over the conversation.
7. Aftercare = scheduled D1/D7 (day and hour should be customizable) followup messages + AI handles followup chat with safety guardrails (zero diagnosis language).
8. Analytics deferred — placeholder folder + README only so we don't forget.

## Approaches Considered

### Approach A: Single-tenant first, refactor for multi-tenant later
~4–6 weeks. Hardcoded for clinic #1. Faster first demo but eats a 3–4 week refactor when clinic #2 onboards — exactly when momentum matters most.

### Approach B: Multi-tenant SaaS from day 1 (CHOSEN)
~6–8 weeks. `org_id` on every table, per-clinic config in DB, per-clinic Line OA credentials, role-based admin UI. Clinic #2 onboards in a day once clinic #1 is dialed in. Unlocks the cofounder's distribution channel.

### Approach C: Chatwoot backbone + custom AI worker
~4–5 weeks. Skips inbox/escalation UI work but couples to Chatwoot's roadmap, fights its human-agent-first design assumptions, and the Line channel quality on Chatwoot is unverified for the Thai market.

## Recommended Approach

**Approach B**, with these decisions locked (so engineering can start week 1 without re-debating):

- **App:** Next.js (App Router) + TypeScript. Single deployment hosts both the customer-message pipeline (Line webhook) and the staff-facing admin UI.
- **DB:** Postgres on Supabase, Thailand? region. `pgvector` extension for embeddings. **Tenant isolation = Postgres Row-Level Security** with `org_id` claim in Supabase JWT. RLS policy template lives in `/db/policies/`. (Decision: RLS over app-side guards — fewer footguns, foundational decision can't drift downstream.)
- **AI brain:** Pluggable LLM provider layer — single internal interface, swap OpenAI / Anthropic / Google at runtime based on per-clinic config. Default new clinic gets Claude (top Thai fluency) selected, owner can change.
- **Embedding model:** **Cohere `embed-multilingual-v3` (1024 dim).** Locked in schema. Can also be configured to other providers, e.g. OpenAI. 
- **pgvector index strategy:** Composite `(org_id, embedding)` with HNSW per-tenant or ivfflat with `org_id` partial indexes — pick the simpler ivfflat path for v0. Per-clinic knowledge corpus cap soft-limit: 100MB / ~10k chunks. Revisit if a clinic blows past that.
- **Channel adapter:** Line Messaging API webhook → normalized internal `Message` type → bot pipeline. The adapter interface is what lets FB / WhatsApp / IG plug in later without core rewrites.
- **Scheduler:** **Inngest.** Thailand-friendly, free tier covers v0, durable execution and retry built in. Used for D1/D7 aftercare and broadcast send.
- **Scheduled message reliability:** `scheduled_messages` table with `(clinic_id, customer_id, kind, scheduled_for, idempotency_key)` and status state machine (`pending → sending → sent | failed → dlq`). Max 3 retries with exponential backoff. Idempotency key prevents double-sends on retry.
- **Admin UI:** Same Next.js app. **Two roles in v0: `owner` and `staff`** (manager role deferred until a clinic actually asks for it). Tabs: Inbox (escalated chats + take-over), Knowledge (upload + manage docs), Bookings (intent queue), Broadcasts (compose + send), Settings (LLM provider, brand voice, capacity rules, Line OA credentials, retention, push-quota readout, staff invites).
- **Auth:** Supabase Auth. `org_id` claim in JWT, used by RLS.
- **Secrets:** Per-clinic LLM API keys and Line OA channel credentials encrypted at rest in **Supabase Vault**. Master key stored in Vercel env var. Rotation runbook documented in `/docs/secrets-rotation.md`.
- **Hosting:** Vercel (app) + Supabase Singapore (DB + Vault) + Inngest (scheduler). All Singapore-region or Singapore-aware.

**Build order (8 weeks including buffer):**

1. **Week 1:** Multi-tenant DB schema with RLS policies, auth, clinic onboarding flow, Line OA credential capture (with signature verification), deploy skeleton to Vercel + Supabase Singapore. Resolves blockers Q4 + Q6 first (see Open Questions).
2. **Week 2:** Line webhook ingestion (signed + async-acked under 1s). Knowledge upload + pgvector indexing + Cohere embedding.
3. **Week 3:** LLM provider abstraction. Chat pipeline: receive → RAG retrieve → LLM reply with brand voice → escalate-or-respond decision. PDPA consent message on first inbound.
4. **Week 4:** Admin inbox (see active/escalated chats, take over, send manual reply). Brand voice + LLM provider configuration UI in Settings.
5. **Week 5:** Booking intent capture + booking queue UI. Per-clinic capacity rule fields (informational v0, no auto-enforcement). Aftercare scheduler infrastructure: `scheduled_messages` table + Inngest jobs + state machine.
6. **Week 6:** Aftercare safety guardrails (deny-list prompt + keyword tripwires + LLM-judge classifier + `audit_events` table). 50-turn labeled eval set built with clinic #1. Simple broadcast composer + push-quota readout. **Clinic #1 cutover happens at end of week 6.**
7. **Week 7:** Polish + bug squash + clinic #1 stabilization + demo prep. DSAR endpoint + retention TTL. Analytics folder placeholder + README listing the events to track later.
8. **Week 8 (buffer):** Whatever the previous 7 weeks didn't predict. Singapore region quirks, Line webhook edge cases, scheduler retries — there will be one of these. Plan for it.

## Open Questions

**Week-0 blockers (resolve before week 1 starts — they shape the schema):**

- **Q4 — Booking format:** Exact fields to capture (date, time, treatment, customer name, phone, notes)? Get clinic #1's actual intake form. Without this, the bookings table is guesswork.
- **Q6 — Customer identity:** Line user ID alone as primary key, or capture phone/email at booking time for cross-channel continuity later? Phone is the practical choice in Thai SMB context; recommend that. Confirm with clinic #1.

**Non-blocking (can be answered during weeks 1–4):**

1. **Brand voice configuration UX:** Free-form prompt textarea, or structured (tone / formality / register sliders + sample dialogues)? Recommend free-form text plus 3–5 sample dialogues per clinic for in-context examples — gives best results with current LLMs and gives the clinic a tangible "show, don't tell" lever.
2. **Knowledge ingestion structure:** Single text dump per category, or structured docs (treatments / devices / FAQ / brand stories)? Recommend structured — RAG recall is meaningfully better and clinics can edit one section without re-embedding the whole base.
3. **Escalation criteria tuning:** LLM confidence threshold, keyword triggers, or staff-defined flags? All three layered; staff-defined trigger phrases as the editable knob in Settings.
4. **Multi-channel timing:** Line-only at launch, but how aggressively does FB/IG/WhatsApp enter the roadmap? Affects how much abstraction effort to invest in the channel adapter in v0 vs. defer to v1.

## Success Criteria

- Clinic #1 cutover from human-only Line replies to Yuna handling inquiries with staff escalation, **end of week 6**. Week 7 stabilizes; week 8 is buffer.
- 80%+ of inbound Line inquiries answered by Yuna without escalation, within 60 seconds of receipt (measured async after the reply-token window).
- Booking intent capture rate ≥ 50% of inquiries that reach a booking topic in the conversation.
- Aftercare followups sent at D1/D7 with >90% delivery (per scheduled_messages state machine) and aftercare replies passing the LLM-judge guardrail with zero diagnosis violations on the 50-turn eval set (auditable in `audit_events`).
- Clinic #2 onboarding from zero to live in ≤ 1 working day. This is the cofounder-network test — if this fails, the architectural premise of Approach B was wrong.

## Distribution Plan

- **For clinic staff (the buyer):** Web app at a stable URL (e.g. app.yuna.ai). No install. Clinic owner signs up, configures Line OA, uploads knowledge, sets brand voice, picks LLM provider, invites staff.
- **For end customers:** They never see Yuna directly. They message the clinic's Line OA as usual; Yuna handles the backend.
- **Pricing model (TBD):** Per-clinic SaaS monthly, likely tiered by message volume or active staff seats. Decide once you have 3 paying clinics — pricing in the dark is worse than pricing on data.

## Dependencies

- Line Official Account API access for clinic #1 (clinic provides Channel ID + Channel Secret + access token).
- LLM provider API keys (OpenAI / Anthropic / Google) — clinic-supplied per their choice in v0; platform-supplied with markup as a later pricing option.
- Cofounder agreement formalized before clinic #2 ships (equity, IP ownership, decision rights). See assignment.

## The Assignment

**Before you write code, sit behind clinic #1's front desk for 2 hours during peak Line inquiry time.** Watch the staff reply. Note the 5 most common question types, the average response time, the messages that get ignored, and the 1 question that always gets escalated to the senior staff member. Bring those notes back — they shape what gets built in weeks 2–3 and they prevent you from building features the clinic doesn't actually need.

**Then: get the cofounder relationship in writing before clinic #2 ships.** Equity split, IP ownership, decision rights. You then added back exactly the right things (booking queue, broadcast) without re-bloating, and pushed aftercare back in only when it became nearly-free given the architecture. That's product-shaped thinking.
- "this is big" was your first reaction when the cofounder context came in. That wasn't excitement filler — you immediately re-prioritized the conversation away from feature exploration and toward speed. Founders who can re-rank under new info are the ones who survive contact with users.

## Screen Inventory & Information Hierarchy

The admin app has 5 tabs. Each screen below lists purpose, what the user sees first/second/third, primary CTA, and any edge that affects layout.

### Inbox (default landing tab, most-used)
**Purpose:** Staff sees live + escalated chats, can take over any thread, can manually reply.
**Information hierarchy:**
1. **Escalated chats list** (left rail) — sorted by escalation timestamp newest-first. Each row: customer Line display name, last message preview (1 line, truncated), escalation reason badge (red = tripwire, amber = low confidence, blue = staff-request), age in minutes.
2. **Active chat thread** (center column) — full message history. Yuna's replies tagged with a subtle "Yuna" badge; staff replies tagged with staff initials. Auto-scrolls to latest. RAG sources for each Yuna reply shown inline (collapsible).
3. **Customer context** (right rail) — Line user ID, phone (if captured), previous bookings, language, Yuna's inferred conversation goal, audit log link.

**Primary CTA:** "Take over chat" (top-right of active thread). Pressing it puts Yuna silent for that thread; staff types in the composer at the bottom.

**Secondary actions:** "Resolve & return to Yuna" (post-takeover), "Flag for review" (escalates to owner), "Add to knowledge" (right-click a good Yuna reply → drops into Knowledge as a sample).

### Knowledge
**Purpose:** Owner/staff uploads, structures, and edits the clinic-specific content Yuna grounds its replies on.
**Information hierarchy:**
1. **Knowledge tree** (left rail) — hierarchical: Treatments / Aftercare / FAQ / Brand stories / Pricing. Each node shows chunk count and last-edited date.
2. **Editor** (center column) — Markdown editor with live preview. "Reindex" button appears when content is dirty.
3. **Metadata panel** (right rail) — language(s), tags, "active in chat" toggle (lets owner stage drafts without affecting live replies), embedding model + dim (read-only).

**Primary CTA:** "Add document" (top-left, prominent). Modal asks for category, language, optional tags, then drops user into the editor.

**Empty state (critical):** A clinic with zero documents sees a 3-step starter: "1. Upload your treatment menu. 2. Upload your top 10 FAQs. 3. Upload one brand-voice sample." Each step has an upload button. **No Yuna replies will go live until step 1 completes — hard gate, not a suggestion.**

### Bookings
**Purpose:** Staff sees inbound booking intents captured from chat; confirms manually.
**Information hierarchy:**
1. **Pending queue** (top) — newest first. Each card: customer name, requested treatment, requested date/time, captured phone, link back to source chat.
2. **Confirmed** (middle) — past 30 days.
3. **Declined / no-show** (bottom, collapsed by default).

**Primary CTA:** "Confirm" (green, on each pending card) or "Reschedule" (opens a template message back to the customer's Line).

**Edge:** Booking intents with ambiguous fields ("next week", "afternoon") get a yellow border and a "needs phone" / "needs time" badge listing what's missing.

### Broadcasts
**Purpose:** Owner composes and sends manual broadcasts (promotions, news).
**Information hierarchy:**
1. **Push-quota readout** (banner, always visible at top) — "Used 4,200 / 10,000 this month — resets in 8 days." Turns amber at 80%, red at 95%.
2. **Composer** (center) — message body with character counter (Line limit 5,000), optional image, target segment (all / last-90-day bookings / staff-defined tag), schedule (now / scheduled).
3. **Past broadcasts** (below) — list with read receipts (if Line OA tier supports) and reply counts.

**Primary CTA:** "Send now" (large, bottom-right). Disabled until a target segment is selected.

**Edge:** If a broadcast would exceed remaining monthly push quota, Send is disabled with a tooltip explaining how many recipients fit in the remaining quota.

### Settings
**Purpose:** Owner configures everything that affects Yuna's behavior, billing, and team.
**Information hierarchy — nine independently-saving sections, each its own collapsible card. DO NOT cram into one scrolling page:**
1. **Clinic profile** — name, address, hours, languages spoken.
2. **Line OA** — channel ID + secret (encrypted, last-4 visible only), webhook URL (read-only), "Test signature" button.
3. **AI brain** — LLM provider picker (OpenAI / Anthropic / Google), model dropdown, embedding provider (locked to Cohere v0 with "request override" link), temperature slider.
4. **Brand voice** — free-form prompt textarea + 3–5 sample dialogue pairs. See Unresolved.
5. **Capacity rules** — informational fields ("we handle X bookings/day per treatment"); surfaces in chat to prevent overbooking talk. v0 = informational only.
6. **Aftercare schedule** — D1/D7 toggles, hour-of-day picker (clinic timezone), template messages per language.
7. **Privacy & retention** — conversation TTL (default 24mo), DSAR export button, sub-processor disclosure (auto-generated from current AI brain selection).
8. **Team** — staff invites, role assignments (owner / staff), audit log access.
9. **Billing** — current plan, message volume meter, payment method.

**Primary CTA per section:** "Save changes" — each section saves independently; no global save button.

**Edge:** Changing the LLM provider shows a confirmation modal: "This affects all future replies. Active conversations finish on the current provider." Save is logged to `audit_events`.

## Visual Hard Rules & AI-Slop Risk Audit

There is no visual `DESIGN.md` yet. To prevent the admin UI from defaulting to AI-slop patterns, lock in these hard rules now.

**Classifier:** the admin app is **APP UI** (dense, task-focused, dashboard-adjacent). The customer-facing surface is Line itself — not designed by us beyond message text.



**Hard rules for the admin UI:**

# Engineering Review Appendix

Added by `/plan-eng-review` on 2026-05-25. The design-review appendix above (8/10) covers UI/UX and visual rules; this appendix covers architecture, code quality, tests, performance, and engineering decisions. **22 decisions resolved (D1–D22), 5 outside-voice findings unresolved, 16 engineering implementation tasks (ET1–ET16).** Each decision has a D-ID for traceability in commits and PRs.

## Engineering Review Additions

These decisions resolve specific architectural gaps in the design body above (lines 1–172). Each links to a decision ID (D1–D22).

### Tenant isolation (D1)

**Server-side workers (Inngest functions) authenticate to Postgres via `SET LOCAL app.current_org_id = '<uuid>'` at job start.** RLS policies use `current_setting('app.current_org_id')::uuid` instead of (or in addition to) the JWT claim. This makes RLS the single source of truth for tenant isolation — code cannot forget to filter. Worker invocations without `SET LOCAL` get zero rows back (fail-safe).

User-facing requests continue to use the Supabase JWT `org_id` claim path (unchanged from original plan).

### pgvector index strategy (D2)

**Single shared `ivfflat` index on the embedding column + `WHERE org_id = ?` pre-filter** in every retrieval query. Tune `lists ≈ sqrt(N)` where N is the total chunk count across all clinics. NOT per-tenant partial indexes (didn't scale, would fight the ≤1-day-onboarding metric).

Migration path: switch to HNSW with `hnsw.iterative_scan = 'strict_order'` (pgvector ≥ 0.7) once a single clinic's corpus exceeds ~20k chunks or recall complaints surface.

### Reply policy (D3)

**Reply-token-only path with hard 50s budget** for inbound chat. Reply API used exclusively. If `embed + retrieval + LLM` cannot complete in 50s (10s buffer for network → 60s Line cap), fall back to a quick Thai ack ("พิมพ์อยู่นะคะ ทีมงานจะตอบกลับเร็วๆ นี้") and route to staff escalation inbox. Push API is reserved for broadcast + scheduled aftercare. Zero push-quota cost for ordinary chat.

### Judge classifier (D4)

**Small dedicated fast model (Claude Haiku OR Gemini Flash, hard-coded across all clinics) acts as the aftercare judge, running pre-send.** Independent of the clinic's chosen generator (breaks circular trust). On timeout / rate-limit / error / 5xx: do NOT send the AI reply — route to staff escalation. Fail-safe by construction.

Judge prompt is stable across all clinics → marked as a prompt-cache breakpoint on Anthropic to halve latency + cost.

### Observability (D5)

**Wire Sentry + Vercel Observability + Inngest DLQ alerts in week 1.** Alert rules: webhook signature failure rate >0, DLQ depth >0, judge classifier error rate >5%, scheduled_messages stuck in `sending` >24h. Email alerts v0 (Slack webhook later). Plan for migration to Datadog / Better Stack once revenue justifies.

### Secrets storage (D6)

**Drop the "master key in Vercel env" line.** Use Supabase Vault's built-in server-managed encryption directly. Per-clinic LLM API keys + Line OA channel credentials encrypted at rest via Vault. Decryption in-process via the standard Supabase Vault client.

v1 migration path documented: move wrapping key to AWS KMS or GCP KMS once revenue justifies (~$1/mo + audit log + rotation).

### Webhook replay protection (D7)

**Inngest idempotency key = Line's `webhookEventId`** on every event emitted from the webhook handler. Inngest dedupes within its default 24h window. No extra `processed_webhook_events` table to maintain.

### Image handling (D20, customized)

On image / sticker / audio / video inbound, AI politely replies in Thai (per-clinic brand voice):

> "ขอโทษนะคะ ระบบยังดูรูปไม่ได้ค่ะ จะให้ถามข้อความได้ไหมคะ หรืออยากให้ทีมงานช่วยดูรูปคะ?"

Two paths offered to the customer:
1. **Reformulate as text** → AI handles normally.
2. **Send to human staff** → escalation queue.

Customer picks via free-text reply. Heuristic intent detection (does reply contain "ส่ง", "ทีมงาน", "พนักงาน", "human", "staff" → escalate; anything else → treat as text turn). Does NOT require multimodal LLM. Reduces escalation flood vs the mute-fallback in the original plan.

### Performance optimizations (D16, D17)

- **Embedding cache:** Normalized-query LRU on inbound (in-memory per worker; Redis-Upstash later). Knowledge ingest uses Cohere batch API (96 docs/call). Reduces inbound embed cost ~70% and ingest time from ~60min to ~5min for 5k chunks.
- **LLM prompt caching (Anthropic path, week 3; Google week 4):** system + brand-voice + safety prompt marked as cache breakpoint. ~90% reduction in cached-token cost + 50–150ms latency improvement per reply. Judge prompt also cached (stable across calls).
- **Aftercare concurrency:** Inngest `concurrency` directive — max 5 concurrent jobs per clinic, max 20 globally per LLM provider, max 10 per Line OA push channel. Random 0–60s jitter added to `scheduled_for` so 9:00 customers spread to 9:00–9:01.

DPA template addresses cross-border transfer (Singapore↔Thailand) — PDPA adequacy is well-established.
